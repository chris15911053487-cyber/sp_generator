# 需求澄清问题改进设计

日期：2026-07-15

## 背景

当前澄清阶段存在三个问题：
1. **问题编号跳跃**：LLM 自行编号，出现"问题1→问题3→问题5"的不连续现象。
2. **一次问多个问题**：`CLARIFY_PROMPT` 虽写了"一次只问一个最关键的问题"，LLM 不严格遵守，有时一次列多个子问题，或把设计方案当澄清问题问。
3. **问题个数不受控**：现有 `user_count >= 6`（含原始需求消息）才强制进设计，实际澄清约 5 轮，但 LLM 可能不收敛。

## 目标

- 编号连续（Q1、Q2、Q3...），由系统控制而非 LLM。
- 严格一次只问一个问题（带选项）。
- 最多 5 个问题，可少于 5 个（LLM 判断信息充足即提前结束）。
- 既保质量又不让用户烦：未澄清的部分在 design 阶段显式假设兜底；最后一问允许用户顺便补充。

## 设计

### 1. `clarify_node`（`app/agent/nodes.py`）

**系统控制编号与轮数**：
- `AgentState` 新增 `clarify_count: int` 字段，记录**已问问题数**（初始 0）。
- 每轮即将问第 `clarify_count + 1` 个问题；把该编号注入 prompt，LLM 只输出问题正文，不含编号。
- 前端展示与 `requirements` 记录时，由系统拼接 `Q{clarify_count+1}: {问题正文}`。
- 开头判断：`clarify_count >= 5`（已问满 5 个）时，强制返回 `mode=design`，不再提问。
- `clarify_count == 4`（即将问第 5 个，即最后一问）时，激活 `last_question_hint`。
- 每问完一轮，`clarify_count += 1`。
- LLM 回复含 `INFO_SUFFICIENT` 时，提前结束（`clarify_count` 可能 < 5）。

**一次只取一个问题**：
- LLM 若在一次回复里输出多个问题，只截取第一个问题（按行或按"问题/Q"标记切分）作为本轮提问，其余丢弃。

**保留原有强制兜底**：现有 `user_count >= 6` 逻辑保留作为外部安全网（防止 `clarify_count` 状态丢失时仍能收敛）。

### 2. `CLARIFY_PROMPT`（`app/agent/prompts.py`）

改写为强约束：
- 明确"只提出 1 个问题，不要自行编号（系统会自动编号），不要一次列多个问题"。
- 问题格式：题干 + 选项 A/B/C（适合选择题时）。
- 信息充足时回复 `INFO_SUFFICIENT` + 需求摘要。
- 新增模板变量 `{last_question_hint}`：
  - 当 `clarify_count == 4`（即将问第 5 个，即最后一问）时，值为"如还有其他需求想补充，可直接一并说明；否则回复选项即可"。
  - 否则为空字符串。
- 模板将该提示附加在最后一问的选项之后，不单独成题。

### 3. `DESIGN_PROMPT`（`app/agent/prompts.py`）

新增显式假设清单要求：
- 方案末尾必须包含"## 我的假设"小节，逐条列出所有未在澄清中明确、但 SP 逻辑依赖的决策（如"假设作废发票=排除""假设科目范围=全量"）。
- 用户在确认 design 时一眼核对，不对则修改，作为质量兜底。

### 4. 答案解析（`clarify_node`）

- 用户回答里提取选项字母（如 `A`/`B`）+ 附加补充文字，一并拼入 `requirements`（`Q{n}: ... \nA: {选项} {补充}`）。
- 用户只回复选项时，补充为空，无需回复"无"。

## 不做（YAGNI）

- 不做"批量选项表格"（与"一次一个"需求冲突）。
- 不做问题类型的硬分类（维度/范围/处理），交由 LLM 基于需求自行判断。
- 不动态调整上限（固定 5）。

## 验证

- 编号连续：跑一次完整澄清流程，确认 Q1→Q2→Q3... 不跳跃。
- 一次一个：LLM 输出多问题时被截断为 1 个。
- 上限 5：到第 5 问后强制进 design。
- 提前结束：简单需求 LLM 回复 `INFO_SUFFICIENT`，`clarify_count < 5` 即进 design。
- 最后一问补充：第 5 问末尾含补充提示，用户可顺便补充。
- design 假设清单：设计方案含"## 我的假设"小节。
