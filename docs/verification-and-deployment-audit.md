# SQL 校验与部署功能审查结论

> 审查日期：2026-07-19
> 审查范围：校验 SQL 生成、单个 SP「校验」、「校验+保存」、「一键预检」及「一键部署」的校验逻辑
> 审查方式：静态代码审查；未连接、执行或修改真实 SQL Server

## 一、总体结论

当前系统存在多条名称相近但含义不同的“校验”路径，各路径的校验范围、副作用和可信度并不一致。

最重要的结论如下：

1. SP 生成后的自动校验会把 SP 实际部署到 SQL Server，再执行数据对比。因此，页面上的「一键部署」不一定是第一次部署。
2. 页面「校验」和「校验+保存」通常只能确认校验 SQL 可以执行，不一定真正执行并对比当前 SP。
3. 「一键预检」和「一键部署」的服务端门禁都只做 `SET PARSEONLY` 语法解析，不检查业务校验结果。
4. “业务通过”“已校验”“预检通过”目前均不能直接等同于“当前编辑代码已完成真实业务验证，可以安全部署”。

## 二、功能对照

| 功能 | 当前实际行为 | 保存内容 | SQL Server 操作 |
| --- | --- | --- | --- |
| 校验 SQL 生成 | LLM 根据已确认设计和 SP 代码生成独立查询 SQL | 保存校验 SQL、对比列和参数 | 随后的自动校验会执行查询，并可能部署 SP |
| 「校验」 | 检查单个 SP 语法，执行数据库中已保存的校验 SQL；满足条件时才对比 SP 结果 | 不保存 SP 代码；会保存参数默认值和校验结果 | 查询数据库；一般不部署 |
| 「校验+保存」 | 先保存 SP 代码，再执行与「校验」相同的逻辑 | 保存 SP 代码、参数默认值和校验结果 | 查询数据库；不部署新代码 |
| 「一键预检」 | 对当前会话全部已保存 SP 做 `PARSEONLY` 语法检查 | 更新 `syntax_valid` | 连接 SQL Server，不部署 |
| 「一键部署」 | 再次检查全部 SP 语法，通过后逐个部署 | 更新部署状态和部署时间 | 正式部署 |

## 三、校验 SQL 生成

### 当前流程

1. 从已确认的设计方案中提取每个 SP 的校验名称、校验方式和对比列。
2. 将设计方案和 SP 代码提交给 LLM。
3. LLM 返回独立的 `SELECT` 校验 SQL、对比列和测试参数。
4. 将结果保存到 SQLite 的 `verify_queries`。
5. SP 与校验 SQL 生成完成后，工作流自动进入校验节点。

主要代码：

- `app/agent/nodes.py`：`_extract_verify_logic`、`_generate_verify_sql_for_sp`、`generate_node`
- `app/agent/prompts.py`：`VERIFY_SQL_PROMPT`
- `app/agent/graph.py`：`generate -> verify` 工作流边

### 当前风险

- 生成校验 SQL 时不探测真实表结构，主要依赖设计方案、SP 代码及 LLM 的 SAP B1 知识。
- 单个 SP 的校验 SQL 生成异常只写日志，整体生成流程仍然继续。
- 如果校验 SQL 解析结果为空，SP 可能没有任何关联校验 SQL。
- 校验阶段的 `business_ok` 默认值为 `True`；当校验 SQL 列表为空时，可能出现没有业务校验却显示业务通过的情况。
- 校验 SQL 执行失败时，自动校验流程会调用 LLM 修复校验 SQL，最多两轮。

## 四、生成后的自动校验

生成完成后，工作流自动进入 `verify_node`，实际流程为：

```text
生成 SP
  -> 生成校验 SQL
  -> SP 语法检查
  -> 实际部署 SP
  -> 执行 SP
  -> 执行校验 SQL
  -> 对比指定列
  -> 失败时最多自动修复两轮
```

### 关键副作用

只要语法检查通过，`verify_node` 就会调用 `deploy_procedure()`。这会修改真实 SQL Server，不是纯只读校验。

自动校验部署成功时会短暂把状态设置为 `deployed`，但校验结束后又把状态覆盖成 `verified` 或 `verify_failed`。因此可能出现：

- SQL Server 中已经存在新 SP；
- 页面和 SQLite 状态却不是 `deployed`；
- 用户误以为点击「一键部署」后才会首次部署。

主要代码：`app/agent/nodes.py` 中的 `verify_node`。

## 五、「校验」按钮

前端入口：`app/templates/index.html` 中的 `verifySp()`。
后端入口：`POST /api/verify/sp/{sp_id}`，实现在 `app/routes/verify.py`。

### 当前行为

1. 从当前 SP 编辑器获取代码。
2. 收集页面参数。
3. 使用 `SET PARSEONLY ON` 检查代码语法。
4. 从 SQLite 加载已保存的校验 SQL并逐条执行。
5. 只有读取到的 SP 状态为 `deployed` 时，才执行 SP 并对比结果。
6. 更新 SP 的语法、业务状态，保存参数默认值和校验结果。
7. 不保存当前 SP 编辑器代码。

### 当前风险

- 自动校验结束会把状态从 `deployed` 覆盖为 `verified`，后续点击「校验」时往往不会执行 SP，只检查校验 SQL 能否执行。
- SP 未部署时，校验 SQL执行成功便会判定该项通过，并提示“仅验证校验 SQL 可执行”。
- SP 执行失败时，当前逻辑仍会将校验 SQL判定为通过，只记录无法对比的警告。
- 没有校验 SQL 时，`business_ok` 仍可能为 `True`。
- 页面允许编辑校验 SQL，但该按钮使用的是 SQLite 中已保存版本，不使用编辑器中的未保存内容。
- 虽然按钮名是「校验」，提交的参数值会被保存为新的默认参数。
- 如果传入的是未保存 SP 新代码，而数据库状态仍是 `deployed`，语法检查可能针对新代码，数据对比却执行 SQL Server 中的旧 SP，形成混合版本校验。

## 六、「校验+保存」按钮

前端入口：`app/templates/index.html` 中的 `saveAndVerifySp()`。

### 当前行为

```text
保存 SP 编辑器代码到 SQLite
  -> 调用单个 SP 校验接口
  -> 保存参数默认值和校验结果
```

它与「校验」的主要区别只是先保存 SP 代码。

### 当前风险

- 当前顺序是先保存、后校验；即使 SQL 无效，也已经覆盖 SQLite 中原有草稿。
- 前端没有检查保存接口的 HTTP 状态和业务结果，保存失败后仍可能继续校验。
- 不保存校验 SQL 编辑器中的修改。
- SP 代码区域只有展开后才初始化 CodeMirror。若未展开代码区域就点击「校验+保存」，前端取得的代码可能是空字符串，并可能将数据库中的 SP 代码覆盖为空。
- 保存 SP 新代码不会自动重新部署；如果原记录以前部署过，后续数据对比仍可能执行 SQL Server 中的旧版本。

## 七、「一键预检」

前端入口：`precheckDeploy()`。
后端入口：`POST /api/deploy/precheck/{session_id}`，实现在 `app/routes/deploy.py`。

### 当前行为

- 从 SQLite 读取会话下全部 SP。
- 对每个 SP 执行 `check_syntax()`。
- 更新 `syntax_valid`。
- 全部通过后，由前端启用「一键部署」按钮。

### 不检查的内容

- 是否存在校验 SQL；
- 校验 SQL 是否能够执行；
- SP 与校验 SQL的数据是否一致；
- SP 当前是否为 `verified`；
- 参数是否完整或有效；
- 页面上尚未保存的修改；
- 表、列、对象和权限是否在实际执行时全部有效。

因此，“一键预检通过”的准确含义是：

> SQLite 中保存的全部 SP 文本通过了 SQL Server `PARSEONLY` 语法解析。

它不表示业务正确，也不表示一定能部署成功。

## 八、「一键部署」中的校验

前端入口：`deployAll()`。
后端入口：`POST /api/deploy/{session_id}`，实现在 `app/routes/deploy.py`。

### 当前行为

1. 重新从 SQLite 读取全部 SP。
2. 在服务端重新对每个 SP 做语法检查，不依赖前端预检结果。
3. 任一 SP 语法失败时阻止本次部署。
4. 全部语法通过后逐个调用 `deploy_procedure()`。
5. 单个 SP 部署成功后更新其 `status=deployed` 和 `deployed_at`。

### 当前风险

- 部署门禁只检查语法，不检查 `business_valid` 或校验 SQL 状态；业务校验失败的 SP 仍可部署。
- 多个 SP 按顺序独立部署，没有整体事务，可能发生部分成功、部分失败。
- 普通 `CREATE PROCEDURE` 的部署策略是先删除同名旧过程，再创建新过程；如果创建失败，旧版本可能已经被删除。
- 部署接口内部语法检查失败时只返回通用消息，未返回详细失败列表；现有前端可能显示为空白结果。
- 前端要求先预检才能启用按钮，但直接调用部署 API 可以绕过页面按钮；后端只保留语法门禁。

## 九、状态含义说明

现阶段应按以下方式理解页面状态：

| 页面状态 | 当前准确含义 | 不能据此推断 |
| --- | --- | --- |
| 语法通过 | `PARSEONLY` 没有报告语法错误 | 表列一定存在、权限正确、一定能创建或执行 |
| 业务通过 | 所有关联校验 SQL 被当前逻辑判定通过 | 一定执行过 SP、一定完成真实数据对比 |
| 已校验 | 当前记录状态为 `verified` | SQL Server 中尚未部署或已经部署 |
| 预检通过 | 当前保存的全部 SP 通过语法解析 | 业务正确、校验 SQL 正确、部署一定成功 |
| 已部署 | 最近一次一键部署成功并写入了部署状态 | 当前编辑器中未保存或未重新部署的代码已生效 |

## 十、建议改进优先级

### P0：阻止意外修改真实数据库

1. 移除生成/自动校验阶段的真实部署副作用。
2. 使用隔离的临时对象、专用测试库或可回滚事务完成业务验证。
3. 明确区分“验证用临时部署”和“正式部署”。

### P0：消除错误通过

1. 没有校验 SQL 时不得默认 `business_ok=True`。
2. SP 未执行、执行失败或没有完成数据对比时，不得显示业务通过。
3. 对比必须明确绑定到与当前编辑代码一致的 SP 版本。

### P1：修正保存行为

1. 「校验+保存」改为先校验，通过后保存，或明确提供“保存草稿”和“验证并保存”两种动作。
2. 获取编辑器内容时同时支持 CodeMirror 和原始 textarea，避免未展开代码区域时保存空字符串。
3. 检查保存接口返回结果，失败时停止后续校验。
4. 保存并使用校验 SQL 编辑器中的修改。
5. SP 代码变化后重置旧的部署/校验状态。

### P1：加强部署门禁

1. 一键部署除语法外，还应要求业务校验通过且校验版本与当前代码一致。
2. 返回完整预检失败详情。
3. 为批量部署增加事务、备份或失败恢复策略。
4. 避免“先删除旧过程、再创建新过程”的不可恢复窗口，优先使用 `CREATE OR ALTER PROCEDURE` 或可靠回滚方案。

### P2：补充测试

建议增加以下自动化测试：

- 校验 SQL 为空时不得业务通过；
- 未部署 SP 不得显示数据对比通过；
- SP 执行失败不得业务通过；
- 未展开编辑器时「校验+保存」不得覆盖为空；
- 校验 SQL 编辑内容能够保存并被实际使用；
- 一键部署拒绝业务校验失败或版本过期的 SP；
- 多 SP 部署部分失败时能够恢复；
- 生成和普通校验流程不会修改正式 SQL Server。
